## Decision 1: Schema Design
**Recommendation:** Option C (badgeConfig with explicit keys + optional custom array)

**Rationale:**
- Keeps strong typing and clear renderer mapping for known badge types.
- Avoids future breaking changes when adding new badge types.
- Allows incremental adoption (promotional + cta first, seasonal later) without contract churn.

**Proposed shape (compatible with current render flow):**
```
badgeConfig: {
  promotional?: { text: string; position: "top-left" | "top-center" | "top-right"; show: boolean; },
  cta?: { text: string; style: "urgent" | "elegant" | "neutral"; show: boolean; },
  seasonal?: { text: string; show: boolean; },
  custom?: Array<{ type: string; text: string; position?: string; style?: string; show: boolean; }>,
}
```

**Minimizing breaking changes:**
- Add `badgeConfig` as optional while keeping existing fields for one release if needed.
- Renderer consumes only `badgeConfig` when present; fallback to defaults otherwise.

## Decision 2: Service Architecture
**Recommendation:** Option A (extend Motor 2) with fallback templates

**Rationale:**
- Motor 2 already owns intent resolution and has all campaign context.
- Single AI call preserves latency budget and reduces operational complexity.
- A separate service adds an extra hop without strong scaling benefit for v2.

**Mitigation:**
- Add deterministic fallback template library if AI output is invalid.
- Log AI badge outcomes for quality review and future template mining.

## Decision 3: Prompt Engineering Risk Analysis
**Assessment:** Prompt-only is not robust enough without strict post-validation.

**Required mitigations (must-have):**
1. Hard validation of length and required fields (Zod + post-checks).
2. Retry once if validation fails; then fallback to templates.
3. Deterministic truncation guard (max chars) with safe suffix trimming.
4. Maintain a small, approved fallback template set per context type.
5. Add test fixtures with 10-20 campaign samples to verify outputs.

**Additional guardrails (nice-to-have):**
- Add a small lexicon filter to block off-brand or risky words.
- Track badge diversity to avoid repetitive outputs.

## Decision 4: Performance & Cost Impact
**Assessment:** Acceptable within current pipeline thresholds.

- New latency: +0.5s to +1s (still far below 20s pipeline budget).
- New cost: +$0.002/campaign (still far below $0.05/campaign budget).

**Optimization note:**
- Keep badge prompt compact; avoid verbose examples.
- Reuse existing campaign input; no extra data fetches.

## Decision 6: Alternative Approaches Considered
- **Separate Badge Service:** Rejected for v2 due to added latency and ops overhead.
- **Template-only:** Rejected; fails requirement for contextual, product-specific text.
- **Hybrid AI -> User edits -> Save template:** Consider for v2.1 as a quality feedback loop.

## Final Recommendation
Proceed with Motor 2 extension using `badgeConfig`, enforce strict validation + fallback templates, and stage seasonal badge in phase 2 to control risk. This meets the user requirement (contextual badges) while keeping pipeline reliability.

## Blockers & Dependencies
- Fallback template set must exist before rollout.
- Tests for badge validation and rendering needed to avoid regressions.
- If seasonal toggle is included, requires DB field and UI update to remain consistent with plan rules.
