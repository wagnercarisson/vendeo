## 📋 CONTEXT — What Needs Assessment

### The Problem
Vendeo's current approach generates artes with **hardcoded badge texts**:
- Always "MEGA OFERTA", "COMPRE AGORA", "50% OFF"
- Generic, not product-specific
- No adaptation to campaign context/instructions
- Creates "bonitas mas frias" (beautiful but cold) visual arts

### User Requirement
> "Quero que cada campanha tenha textos únicos e contextuais, não sempre os mesmos jargões promocionais."

### Proposed Solution
Extend **Motor 2 (Intent Resolver)** to generate contextual badge/CTA texts via AI:
- Motor 2 already uses GPT-5.4
- Motor 2 already receives campaign data (product_name, price, visual_context, description)
- Add 3 badge fields to `CreativeDirection` output
- Use AI to generate product-specific texts: "Última Coca 2L: R$ 5,90" (not "MEGA OFERTA")

### Your Mission
**Validate if this is the right technical approach** — or identify alternatives that might be better.

## 🏗️ PROPOSED ARCHITECTURE

### Current System (Production)

**Motor 2 (Intent Resolver):**
- File: `lib/ai/intent-resolver/service.ts`
- Model: GPT-5.4
- Input: `CampaignData` + `ImageProfile` + `VisualSignature`
- Output: `CreativeDirection` (directionType, mood, priceEmphasis, visualIntensity)
- Current latency: ~2-3s
- Current token usage: ~800-1000 tokens

**Motor 4 (Renderer):**
- File: `lib/ai/renderer/canvas.ts`
- Uses hardcoded `RENDERER_COLORS` const
- Uses hardcoded `PROMOTIONAL_BADGES` const (if exists)
- Composes final image with Sharp

### Proposed Extension

**Extend `CreativeDirection` schema:**

```typescript
// lib/ai/intent-resolver/contracts.ts

export const CreativeDirectionSchema = z.object({
  // Existing fields...
  directionType: z.enum(['hero', 'frame', 'split-dynamic', 'overlay', 'stacked']),
  mood: z.enum(['clean', 'aggressive', 'playful', 'premium', 'urgency']),
  priceEmphasis: z.number().min(0).max(10),
  visualIntensity: z.number().min(0).max(10),
  
  // NEW: Badge fields
  promotionalBadge: z.object({
    text: z.string().max(30), // "Última Coca 2L: R$ 5,90"
    position: z.enum(['top-left', 'top-center', 'top-right']),
    show: z.boolean(),
  }).optional(),
  
  ctaBadge: z.object({
    text: z.string().max(25), // "Garanta já" or "Reserve sua garrafa"
    style: z.enum(['urgent', 'elegant', 'neutral']),
    show: z.boolean(),
  }),
  
  seasonalBadge: z.object({
    text: z.string().max(30), // "Natal 2026" or "Especial de Natal"
    show: z.boolean(),
  }).optional(),
});
```

**Extend Motor 2 prompt:**

Add `<badge_generation>` section with rules:
- Parse `description` field for intent keywords ("última unidade", "exclusivo", "promoção")
- Generate contextual badge texts (product-specific, not generic jargon)
- Match tone to `visual_context` (promotional = urgent, premium = elegant)
- Respect length constraints (promotionalBadge max 30 chars, ctaBadge max 25)

**Update Motor 4 (Renderer):**
- Remove hardcoded `PROMOTIONAL_BADGES` const
- Read badge texts from `CreativeDirection` object
- Use Motor 2 outputs directly

**Estimated impact:**
- +~500 tokens to Motor 2 prompt
- +0.5-1s latency (AI generation time)
- +$0.002/campaign (OpenAI cost)

## 🧠 KEY TECHNICAL DECISIONS TO VALIDATE

### Decision 1: Schema Design

**Option A (Proposed):** 3 separate badge fields
```typescript
promotionalBadge: { text, position, show }
ctaBadge: { text, style, show }
seasonalBadge: { text, show }
```

**Option B:** Badge array for extensibility
```typescript
badges: Array<{
  type: 'promotional' | 'cta' | 'seasonal' | 'custom',
  text: string,
  position?: string,
  style?: string,
  show: boolean,
}>
```

**Option C:** Badge config object
```typescript
badgeConfig: {
  promotional?: { text, position, show },
  cta?: { text, style, show },
  seasonal?: { text, show },
  custom?: Badge[],
}
```

**Your assessment:**
- Which schema is most scalable for future badge types?
- Which is most compatible with Motor 4 rendering logic?
- Which minimizes breaking changes?

---

### Decision 2: Service Architecture

**Option A (Proposed):** Extend Motor 2 (Intent Resolver)
- **Pros:** Reuses GPT-5.4, already has campaign data, single call
- **Cons:** Motor 2 becomes more complex, single point of failure

**Option B:** Separate Badge Generation Service
- **Pros:** Separation of concerns, independent scaling
- **Cons:** Extra API call, latency increase, data duplication

**Option C:** Badge Template Library (AI-assisted selection)
- **Pros:** Consistency, performance (no AI generation), predictable
- **Cons:** Still needs templates, less creative, maintenance burden

**Option D:** Hybrid (AI suggests → User edits → Save as template)
- **Pros:** Best of both (creativity + control), builds library over time
- **Cons:** Extra UI complexity, requires user intervention

**Your assessment:**
- Is Motor 2 the right place for badge generation?
- Should we consider a separate service?
- Is a hybrid approach more robust?

---

### Decision 3: Prompt Engineering Risk Analysis

**Proposed approach:** 100% AI-generated badge texts

**Risks identified:**
1. **Hallucination:** AI generates nonsensical or off-brand texts
2. **Length violations:** AI ignores max 30 chars constraint
3. **Inconsistency:** Same product generates different texts each time
4. **Generic fallback:** AI falls back to "COMPRE AGORA" when uncertain

**Mitigation strategies:**
- Strict prompt constraints with examples
- Few-shot learning (10+ input/output examples)
- Retry logic (max 2 attempts if length violated)
- Fallback to generic templates if AI fails

**Your assessment:**
- Is prompt engineering robust enough for production?
- What additional mitigations should we implement?
- Should we have a fallback template library?
