### **3. Acceptance Criteria (Gherkin Format)**

**AC1: Badge Generation**
```gherkin
GIVEN a campaign with promotional intent and campaign instructions
WHEN Motor 2 (Intent Resolver) processes the campaign
THEN it generates a badgeConfig object with contextual badge text (<30 chars)
AND badge position matches context (promotional → top-left)
AND badge text is product-specific when possible
```

**AC2: CTA Generation**
```gherkin
GIVEN a campaign with urgency intent
WHEN Motor 2 processes the campaign
THEN it generates a CTA text (<25 chars) that reflects urgency
AND CTA text is action-oriented
AND CTA text avoids generic jargon
```

**AC3: Fallback Templates**
```gherkin
GIVEN Motor 2 AI generation fails or produces invalid output
WHEN fallback is triggered
THEN Motor 2 uses fallback templates from FALLBACK-TEMPLATES-MOTOR2-PHASE1.md
AND fallback templates match the campaign context
AND system retries AI generation once before fallback
```

**AC4: Rendering Integration**
```gherkin
GIVEN a campaign with badgeConfig from Motor 2
WHEN Motor 4 (Renderer) renders the campaign image
THEN badge is positioned according to Visual Composition Guide
AND badge color matches context
AND badge text is legible (≥4.5:1 contrast)
AND badge does not occlude product key areas (>15%)
```

### **5. Technical Approach**

**Schema Changes:**
```typescript
// Motor 2 output (Intent Resolver)
export interface IntentResolutionResult {
  intent: CampaignIntent;
  visual_context: string; // Renamed from campaignInstructions
  badgeConfig?: {         // NEW
    badgeText: string | null;
    badgePosition: 'top-left' | 'top-center' | 'top-right';
    ctaText: string | null;
  };
}
```

**Motor 2 Extension:**
- Add `<badge_generation>` section to `lib/ai/intent-resolver/prompts.ts`
- Add Zod validation for `badgeConfig` output
- Add retry logic (max 1 retry) on AI failure
- Add fallback template selection on validation failure
- Keep added tokens <500

**Motor 4 Integration:**
