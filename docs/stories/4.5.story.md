## 🏗️ Technical Approach

### Architecture Decision

**v1 (legado):**
```typescript
// lib/domain/campaigns/service.ts (atual)
generateCampaignContent() {
  buildCampaignPrompt() → callAIWithRetry() → persist text only
}
```

**v2 (novo):**
```typescript
// lib/domain/campaigns/visual-pipeline.ts (novo)
generateCampaignVisuals() {
  Motor 1: analyzeImage() → ImageProfile
  Motor 2: resolveIntent() → CreativeDirection
  Motor 3: composeVariations() → CompositionVariants (4 layouts)
  Motor 4: renderVariations() → PNG uploads (4-6 artes)
  persist URLs + metadata
}
```

**Integration Point:**
```typescript
// lib/domain/campaigns/service.ts (modificado)
generateCampaignContent() {
  // Texto (mantém v1 por enquanto)
  const textOutput = await generateTextContent(...)
  
  // Visual (novo — feature flag)
  if (MOTOR_V2_ENABLED && campaign.product_image) {
    const visualOutput = await generateCampaignVisuals(...)
  }
  
  persist(textOutput, visualOutput)
}
```

### Feature Flag Strategy

```typescript
// lib/constants/features.ts
export const MOTOR_V2_ENABLED = 
  process.env.NEXT_PUBLIC_MOTOR_V2_ENABLED !== 'false'; // default: true
```

**Behavior:**
- `MOTOR_V2_ENABLED === true` (default): Pipeline visual ativada para novas campanhas
- `MOTOR_V2_ENABLED === false`: Apenas geração de texto (v1)
- Campanhas antigas: Mantém comportamento atual (texto apenas)

### Error Handling Strategy

**Principle:** Fail fast with specific error codes, NOT generic errors

```typescript
type PipelineError = {
  motor: 'visual-reader' | 'intent-resolver' | 'visual-composer' | 'renderer';
  code: string; // ex: "IMAGE_LOAD_FAILED", "INVALID_TARGET_BOX", "RENDER_TIMEOUT"
  message: string;
  details?: unknown;
};
```

**Error scenarios:**
1. Motor 1 fails (image load) → Return `{ motor: 'visual-reader', code: 'IMAGE_LOAD_FAILED' }`
2. Motor 2 fails (LLM timeout) → Return `{ motor: 'intent-resolver', code: 'LLM_TIMEOUT' }`
3. Motor 3 fails (validation) → Return `{ motor: 'visual-composer', code: 'INVALID_LAYOUT' }`
4. Motor 4 fails (rendering) → Return `{ motor: 'renderer', code: 'RENDER_FAILED' }`

**User-facing messages:**
- Don't expose internal errors
- Show actionable guidance: "Erro ao processar imagem. Tente fazer upload de outra imagem."

### Idempotency Strategy

```typescript
// Check if campaign already has visual outputs
if (campaign.visual_outputs && !force) {
  return { ok: true, reused: true, outputs: campaign.visual_outputs };
}
```

**Rule:** If campaign already has `visual_outputs` (array of PNG URLs), don't regenerate UNLESS `force=true`.

### Logging Strategy

**Structured logging with trace_id:**

```typescript
const trace_id = crypto.randomUUID();

console.log('[PIPELINE][START]', { trace_id, campaign_id, timestamp: Date.now() });

// Motor 1
console.log('[MOTOR-1][INPUT]', { trace_id, imageUrl, productName, content_type });
const imageProfile = await analyzeImage(...);
console.log('[MOTOR-1][OUTPUT]', { trace_id, imageProfile });

// Motor 2
console.log('[MOTOR-2][INPUT]', { trace_id, imageProfile, campaign });
const direction = await resolveIntent(...);
console.log('[MOTOR-2][OUTPUT]', { trace_id, direction });

// Motor 3
console.log('[MOTOR-3][INPUT]', { trace_id, imageProfile, direction });
const variations = await composeVariations(...);
console.log('[MOTOR-3][OUTPUT]', { trace_id, variations });

// Motor 4
console.log('[MOTOR-4][INPUT]', { trace_id, variations });
const renders = await renderVariations(...);
console.log('[MOTOR-4][OUTPUT]', { trace_id, renders });

console.log('[PIPELINE][END]', { trace_id, duration: Date.now() - start });
```

**Purpose:** Enable debugging in production by searching logs for `trace_id`.

## 🔒 Pipeline Rules

### Rule 1: Motor Execution Order

**MUST execute in sequence (not parallel):**

```
Motor 1 (Visual Reader) → ImageProfile
  ↓
Motor 2 (Intent Resolver) → CreativeDirection
  ↓
Motor 3 (Visual Composer) → CompositionVariants (4 layouts)
  ↓
Motor 4 (Renderer) → PNG uploads (4-6 artes)
```

**Rationale:** Each motor depends on output of previous motor.

### Rule 2: Error Propagation

If ANY motor fails:
1. Stop pipeline immediately (don't proceed to next motor)
2. Return specific error with motor name + code
3. Log full error context with trace_id
4. Do NOT persist partial outputs

### Rule 3: Performance Budget

| Stage | Budget | Cumulative |
|-------|--------|------------|
| Motor 1 (Visual Reader) | <2s | 2s |
| Motor 2 (Intent Resolver) | <3s | 5s |
| Motor 3 (Visual Composer) | <2s | 7s |
| Motor 4 (Renderer) | <3s | 10s |
| **Total p95** | **<10s** | - |

**If exceeded:** Log warning, but don't fail (performance degradation, not error).

### Rule 4: Backward Compatibility

**Campaigns created before v2:**
- Continue using v1 text generation
- No visual outputs
- No breaking changes

**Campaigns created after v2:**
- Use both text (v1) + visual (v2)
- Feature flag controls visual pipeline

### Rule 5: Idempotency

**Check before regenerating:**

```typescript
const { data: existing } = await supabase
  .from('campaigns')
  .select('visual_outputs')
  .eq('id', campaign_id)
  .single();

if (existing?.visual_outputs && !force) {
  return { reused: true, outputs: existing.visual_outputs };
}
```

## 🔥 Production Bug Fix — IMAGE_LOAD_FAILED

**Date:** 2026-04-25  
**Status:** ✅ FIXED  
**Commit:** `02c0592` - fix(visual-pipeline): resolve relative paths to signed URLs [Story 4.5]

### Problem
Após deploy da Story 4.5, geração de campanhas falhava com:
```
Failed to parse URL from stores/283410f0-39a0-44ea-b9d1-2a3a8ddcb3d3/products/image.png
```

### Root Cause
- Upload v2 salva **path relativo** em `product_image_url` (`stores/.../file.webp`)
- Listagem funciona porque usa `getSignedImageUrl()` com healing logic
- Pipeline v2 falha porque `assertImageReachable()` chama `fetch(path)` diretamente sem resolver URL

### Solution (Architectural Decision by @architect)
1. **Path relativo permanece como fonte de verdade** (agnóstico de ambiente, portável)
2. **URL assinada gerada sob demanda** (não persistida)
3. **Resolução em runtime** na pipeline via `getSignedImageUrl()`
4. **Fallback mantido** para campanhas antigas (migration SQL futura)

### Implementation
**File:** `lib/domain/campaigns/visual-pipeline.ts`

```typescript
import { getSignedImageUrl } from "@/lib/supabase/storage-server";

async function assertImageReachable(imageUrl: string, trace_id: string) {
  if (isDataUrl(imageUrl)) return;

  // Resolve path relativo → URL assinada (healing logic)
  const resolvedUrl = await getSignedImageUrl(imageUrl);
  
  if (!resolvedUrl) {
    throw buildPipelineError({
      motor: "visual-reader",
      code: "IMAGE_URL_RESOLUTION_FAILED",
      message: "Erro ao resolver URL da imagem.",
      details: `Failed to resolve: ${imageUrl}`,
    }, trace_id);
  }

  pipelineLog("[PIPELINE][URL_RESOLVED]", {
    trace_id,
    originalInput: imageUrl,
    resolvedUrl: resolvedUrl.substring(0, 100) + "...",
  });

  const response = await fetch(resolvedUrl, { method: "GET" });
  // ... rest of validation
}
```

### Tests Added
- ✅ `resolves relative paths to signed URLs before fetching (v2 support)`
- ✅ `returns a pipeline error when URL resolution fails`
- ✅ **11/11 tests passing** (5 pipeline + 3 integration + 2 compat + 1 perf)

### Impact
- ✅ Desbloqueia Motors 1-4 em produção
- ✅ Compatível com paths (v2) e URLs (v1) via healing
- ✅ Segurança preservada (signed URLs com token de 1h)
- ⚠️ Performance: +50-100ms latência para obter signed URL (aceitável)
