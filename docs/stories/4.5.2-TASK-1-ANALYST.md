## Mission

Analisar outputs reais do **Motor 3 (Visual Composer)** em produção para identificar:
1. **Quais campos** o GPT-5.4 está retornando consistentemente
2. **Qual estrutura** está sendo gerada (vs schema esperado)
3. **Quais campos são úteis** para renderização (Motor 4)
4. **Padrões de erro** nos 100+ Zod validation failures

**Objetivo:** Fornecer dados empíricos para decidir se devemos:
- **Option A:** Atualizar schema para aceitar campos novos (ex: productName, priceBadge)
- **Option B:** Restringir GPT com schema no prompt
- **Option C:** Híbrido (aceitar alguns campos + restringir outros)

---

## Context

### Current State
Motor 3 está **funcional mas subótimo**:
- ✅ Gera 4 variações via fallback (quando Zod validation falha)
- ❌ Logs poluídos com 100+ erros de validação por request
- ❌ Performance lenta (23s de 31s total do pipeline)
- ⚠️ Qualidade das compositions pode ser genérica (não usa campos extras do GPT)

### Schema Atual
**File:** `lib/ai/visual-composer/contracts.ts`

Schema espera estrutura como:
```typescript
{
  compositions: [
    {
      layout: { type: 'centered' | 'hero' | 'minimal' | 'bold' },
      elements: [
        {
          type: 'text',
          content: string,
          style: {
            fontSize: number,
            fontWeight: '400' | '600' | '700' | '900',
            color: string,
            // fontFamily, lineHeight, etc. NÃO estão no schema
          },
          position: { x, y, width, height }
        }
      ]
    }
  ]
}
```

### Exemplos de Erros (Production Logs)
```
[MOTOR-3][ERROR] Zod validation error for output 1:
  - compositions.0.productName: Expected undefined but received 'Coca-Cola 600ml'
  - compositions.0.layers.0.fontFamily: Expected undefined but received 'Montserrat'
  - compositions.0.layers.0.fontWeight: Expected '400' | '600' | '700' | '900' but received 700
  - compositions.0.elements.0.priceBadge: Unrecognized key
  - compositions.0.elements.0.lineHeight: Expected undefined but received 1.2
```

**Hipótese:** GPT-5.4 evoluiu e está retornando campos mais ricos que não estavam no schema original.
