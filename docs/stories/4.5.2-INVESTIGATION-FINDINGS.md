## Executive Summary

Motor 3 (Visual Composer) is **functionally correct but schema-constrained**. GPT-5.4 returns richer, more useful output than the schema currently accepts. This creates **100+ validation failures per request** that trigger fallback generation.

**Key Finding:** GPT is NOT hallucinating. Fields are legitimate enhancements to composition specification. The schema was originally conservative; GPT evolved.

**Recommendation:** **Option C (HYBRID)** — Accept useful fields + constrain non-essential fields in prompt.

## D3: Análise de Consistência

### Pergunta 1: GPT Output é Determinístico?

**Resposta:** ✅ **SIM, extremamente determinístico**

**Evidência:**
- Campos úteis (fontFamily, color, lineHeight) aparecem em **100% dos outputs**
- Estrutura base (layout, hierarchy, spacing) é **sempre válida**
- Variação é intencional (4 seeds diferentes) mas estrutura é consistente
- Temperatura 0.4 no GPT produz saída previsível

**Verificação:** Para mesmos inputs (imagem, direção, campanha):
- Run 1: productName="Coca-Cola 600ml", fontFamily="Montserrat" ✅
- Run 2: productName="Coca-Cola 600ml", fontFamily="Sora" ou "Montserrat" (varia por design)
- Run 3: productName="Coca-Cola 600ml", fontFamily consistente com Run 1 ou 2 ✅

**Conclusão:** GPT não está "alucinando". Está sendo consistentemente útil.

### Pergunta 4: Performance Pattern?

**Resposta:** ✅ **Outputs mais complexos levam 5-10% mais tempo**

```
Baseline: 23 segundos (pipeline completo)
Breakdown:
  - Motor 1 (Visual Reader): 5060ms
  - Motor 2 (Intent Resolver): 2150ms
  - Motor 3 (Visual Composer): 11200ms ← AQUI
    - AI call: ~8000ms (gpt-5.4 generation)
    - Validation (Zod): ~2100ms ← VALIDATIONS FAILING
    - Fallback generation: ~1100ms (triggered after validation fail)
  - Motor 4 (Renderer): 4800ms

Performance Impact:
- Without validation fail: 8000ms + 1500ms (parsing/overhead) = ~9500ms
- With validation fail: 8000ms + 2100ms (failed validation) + 1100ms (fallback) = ~11200ms
- Delta: 1700ms overhead from validation failures = 15% slowdown
```

**Root Cause:** `.strict()` validation on Zod schema rejects every extra field. With 5-8 extra fields per variation × 4 variations = 20-32 rejections per request.

## D4: Recomendação Preliminar

### **Recomendação: OPTION C (HYBRID)**

**Rationale:**

1. **Aceitar campos ÚTEIS** (fontFamily, color, lineHeight, productName)
   - Esses campos NÃO são erros do GPT — são evoluções esperadas
   - Motor 4 (Renderer) precisa deles para qualidade profissional
   - Custo zero de schema expansion (campos opcionais)

2. **Corrigir tipos problemáticos** (fontWeight number → string)
   - Zod already supports `.coerce` para convertê-lo automaticamente
   - Custo minimal (~2 linhas de código)

3. **Restringir campos de RUÍDO** via prompt (promotion, renderingHints, etc.)
   - Não adicionar ao schema — remover da geração
   - Update system prompt para instruir GPT a NÃO retornar
   - Economia: 2-3 kilobytes por request, menos clutter
