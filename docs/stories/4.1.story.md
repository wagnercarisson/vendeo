## 📋 Context

O Vendeo gera artes para campanhas de lojas físicas. Quando o lojista:
1. Escolhe uma aba (**Produtos** | **Serviços** | **Informativos**)
2. Digita o nome do produto (ex: "Coca Cola 600ml")
3. Faz upload de uma imagem

O **Visual Reader v2** é o **Motor 1** da pipeline de composição visual. Ele analisa a imagem e responde: "o que tem aqui, onde está e como está?" — retornando um perfil visual estruturado (`ImageProfile`) que será consumido pelos motores subsequentes.

**Source of Truth:** [4.1-BRIEF.md](./4.1-BRIEF.md)

---

## 🎯 Goal

Implementar o Visual Reader v2 com:
- Contrato de entrada: 3 campos (`imageUrl`, `productName`, `content_type`)
- Contrato de saída: 21 campos estruturados em 6 categorias
- Gate de decisão: campanhas informativas não processam IA
- System prompt otimizado via @prompt-eng
- Cache Supabase para imagens repetidas
- Rota de produção (não sandbox)
- 10 fixtures de teste cobrindo cenários críticos

## 📥 Input Contract (3 fields)

| Field | Type | Required | Source |
|-------|------|----------|--------|
| `imageUrl` | string (URL) | ✅ | Upload → Supabase Storage |
| `productName` | string | ✅ | User input |
| `content_type` | "product" \| "service" \| "message" | ✅ | Derived from active tab |

### Critical Gate Rule
```typescript
if (content_type === "message") {
  return emptyImageProfile(); // Do NOT call AI
}
```

**Reason:** Informative campaigns have no product to locate — processing would waste resources and generate meaningless output.

### About `productName`
- Simple: "Coca Cola 600ml"
- Compound: "Coca Cola 600ml + Batatas Fritas"
- Kit: "Kit c/6 Coca Cola 600ml" or "5 pares de meias sociais"
- Motor infers scene type from name + image (no external derivation)

---

## 📤 Output Contract (21 fields)

### Group 1 — Match Detection (7 fields)

| Field | Type | Purpose |
|-------|------|---------|
| `detected` | boolean | Exact product found? |
| `matchType` | "exact" \| "category_only" \| "none" | Match level with productName |
| `matchedTarget` | string / null | Description of what was found |
| `confidence` | "low" \| "medium" \| "high" | Overall analysis confidence |
| `sceneType` | "single_product" \| "multiple_products" \| "lifestyle_scene" \| "full_scene" \| "unclear" | Scene classification |
| `relevantCount` | number | Number of relevant items found |
| `ignoredElements` | string[] | Items present but not target (future cross-sell) |

**`matchType` Behavior:**
- `exact` → product found with matching brand, format, volume
- `category_only` → similar product found (same category, different brand/format)
- `none` → product not found in image

**`ignoredElements` Value:**  
Future cross-sell potential — Motor 2 may use if items are in store mix.

**`confidence` Behavior:**  
Never blocks generation. System warns user about low confidence, but shopkeeper always decides.

### Group 2 — Spatial (4 fields)

| Field | Type | Purpose |
|-------|------|---------|
| `targetBox` | {x, y, width, height} / null | Normalized coordinates (0-1) of product |
| `targetOrientation` | "vertical" \| "horizontal" \| "square" \| "mixed" \| "unknown" | Product orientation |
| `targetPosition` | "left" \| "center" \| "right" \| "top" \| "bottom" \| "mixed" \| "unknown" | Product position in image |
| `targetOccupancy` | "low" \| "medium" \| "high" \| "full" | How much space product occupies |

**About `targetBox`:**
- Normalized coordinates: x and y between 0 and 1 (image proportion)
- Compound `productName` ("Coca + Batatas"): return **single box** encompassing all items
- `null` only when `matchType === "none"`

**About `targetPosition` and `targetOccupancy`:**
- Fallback when `targetBox` is null
- `targetOccupancy` indicates maneuver room for Motor 2: `low` = lots of space, `full` = no space

### Group 3 — Quality (3 fields)

| Field | Type | Purpose |
|-------|------|---------|
| `imageQuality` | "good" \| "acceptable" \| "poor" \| "unknown" | Overall technical photo quality |
| `visibility` | "clear" \| "partial" \| "obstructed" \| "unknown" | Product visible? Obstructed? Cropped? |
| `framing` | "good" \| "tight" \| "distant" \| "unknown" | Framing: too close, distant, or adequate? |

**Difference between `visibility` and `framing`:**
- `visibility`: Coca is covered by another bottle in front? → `partial`
- `framing`: Coca is very small in corner of photo? → `distant`

Both inform user about photo adequacy and guide Motor 2 in composition.

**User Alerts** (do not block, only inform):
- `imageQuality === "poor"` → low quality photo
- `visibility === "obstructed"` → product covered
- `framing === "distant"` → product too small
- `matchType === "none"` → product not found
- `matchType === "category_only"` → product different from declared

### Group 4 — Background (1 field)

| Field | Type | Purpose |
|-------|------|---------|
| `backgroundType` | "transparent" \| "solid" \| "complex" \| "unknown" | Background type — Motor 2's main decision |

**What each value means for composition:**
- `transparent` → PNG with alpha channel. Can be used freely in any composition
- `solid` → solid color background. Easy to manipulate — Motor 2 can replace or keep
- `complex` → environment, store, usage scene. Use whole image — do not try to crop product

### Group 5 — Crop Potential (4 fields — temporary)

> These fields are kept for data collection and testing. Will be re-evaluated in Story 4.1.1 after real image volume.

| Field | Type | Purpose |
|-------|------|---------|
| `subjectCutoff` | "none" \| "light" \| "moderate" \| "severe" \| "unknown" | Product cut at photo edges? |
| `safeExpansionPotential` | "low" \| "medium" \| "high" \| "unknown" | Space around product to expand crop |
| `focusClarity` | "low" \| "medium" \| "high" \| "unknown" | Specific focus on product (differs from imageQuality which evaluates whole photo) |
| `visualIsolation` | "low" \| "medium" \| "high" \| "unknown" | Product stands out from background? |

### Group 6 — Metadata (1 field — temporary)

| Field | Type | Purpose |
|-------|------|---------|
| `reasoningSummary` | string | Text explanation of what model found. Useful for debug during tests. No use in motors or for user. |

> Candidate for removal in Story 4.1.1.

## 🔔 User Alerts

System **never blocks** generation. Only informs. Shopkeeper decides.

| Condition | Suggested Message |
|-----------|------------------|
| `matchType === "none"` | "Produto não encontrado na imagem. Verifique se a imagem corresponde ao produto." |
| `matchType === "category_only"` | "A imagem parece ter um produto diferente do declarado (`matchedTarget`). Deseja continuar?" |
| `imageQuality === "poor"` | "A qualidade da imagem está baixa. O resultado pode não ser o ideal." |
| `visibility === "obstructed"` | "O produto parece estar parcialmente encoberto na imagem." |
| `framing === "distant"` | "O produto está muito pequeno na imagem. Tente uma foto mais próxima." |
| `confidence === "low"` | "A análise da imagem teve baixa confiança. Revise o resultado." |

## 🤝 Agent Dependency — @prompt-eng

**FIRST TASK:** `@dev` must call `@prompt-eng` **before any implementation code**.

`@prompt-eng` is responsible for composing the `gpt-5.4-mini` (vision) system prompt to be saved in `lib/ai/visual-reader/prompts.ts`.

### What to deliver to @prompt-eng

Pass [4.1-BRIEF.md](./4.1-BRIEF.md) as complete context. Critical points the prompt must cover:

**Motor objective:**
- Analyze image and locate product described in `productName`
- Return structured JSON with 21 output contract fields
- Never invent — base only on what's visible in image

**Matching rules:**
- `exact` → brand + format + volume match `productName`
- `category_only` → similar product found, but not declared — return `targetBox` and describe what was found in `matchedTarget`
- `none` → product not found — `targetBox` must be null

**Compound `productName` rules:**
- Name with "+" or "kit" or "pack" or "c/N" indicates multiple items
- Locate all items from name in image
- Return single `targetBox` encompassing all of them
- If items are at opposite ends of image, box will be large — this is correct

**`targetBox` rules:**
- Normalized coordinates between 0 and 1 (image proportion)
- Must be tight — wrap only product, not entire image
- Mandatory for `matchType=exact` and `matchType=category_only`
- `null` only for `matchType=none`

**`backgroundType` rules:**
- `transparent` → PNG without background / visible alpha channel
- `solid` → single, uniform color background
- `complex` → environment, scene, multiple background objects

**Response format:**
- Pure, valid JSON
- No markdown, no text before or after
- No extra fields beyond the 21 defined

### What @prompt-eng must deliver

- Final system prompt for `lib/ai/visual-reader/prompts.ts`
- Justification of prompt structuring choices
- Few-shot examples (if recommended)
