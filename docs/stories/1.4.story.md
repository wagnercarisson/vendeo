## User Story

**Como** equipe de engenharia do Vendeo,
**Quero** normalizar `info -> message` no dominio de campanha,
**Para** alinhar o runtime ao modelo V2 sem exigir ainda cleanup ou endurecimento final de constraints.

---

## Objective

Isolar a compatibilidade de dominio de campanha em uma story pequena. O foco aqui e apenas ajustar tipos, schemas, mappers e writers canonicos para ler legado e escrever no modelo novo, sem tocar em assets aprovados, weekly plan ou Brand Profile.

---

## Scope IN

- Ajustar tipos, schemas e mappers de `campaigns` para aceitar `info` em leitura legado.
- Expor `message` como valor canonico no contrato interno do dominio.
- Garantir compatibilidade incremental de escrita com a constraint SQL atual, sem reintroduzir drift entre os writers.
- Garantir que writers novos persistam apenas `product | service | message`.
- Usar `legacy_content_type` quando for necessario preservar o valor legado original.

## Scope OUT

- Nao alterar approved assets.
- Nao alterar `weekly_plan_items`.
- Nao implementar fallback de Brand Profile.
- Nao endurecer checks SQL finais.
- Nao executar backfill em massa.

---

## Acceptance Criteria

```gherkin
GIVEN uma campanha legada com `content_type = 'info'`
WHEN a campanha for lida pelo dominio canonico
THEN o contrato interno usa `message`
AND o fluxo atual continua lendo a campanha sem erro
```

```gherkin
GIVEN uma nova escrita de campanha apos esta story
WHEN o writer canonico persistir `content_type`
THEN somente `product`, `service` ou `message` sao emitidos
AND `info` nao volta a ser escrito como nova verdade
```

### Implementation Notes

- O contrato interno do dominio de `campaigns` agora expoe `content_type` canonico como `product | service | message`.
- A leitura legado continua aceitando `info` no schema do banco e o mapper converte esse valor para `message`, preservando `legacy_content_type = 'info'` quando necessario.
- A normalizacao de escrita foi centralizada em `buildCampaignContentTypeWrite`, evitando drift entre criacao de campanha, edicao/approve e duplicacao.
- Enquanto a constraint SQL remota de `campaigns.content_type` ainda aceita apenas `product | service | info`, a borda de escrita persiste `info` para campanhas canonicas `message`, sem alterar o contrato interno do dominio.
- Os formulĂ¡rios internos passaram a trabalhar com `message` como valor canonico, mantendo compatibilidade de borda com query params legados `type=info` via `normalizeCampaignContentType`.
- O contrato de `visual-composition` foi ampliado para aceitar `message` alem de `info`, evitando regressao de runtime em consumidores tipados do campaign domain durante a transicao incremental.
- Nenhum check SQL, backfill, approved assets, weekly plan ou fallback de Brand Profile foi alterado.
