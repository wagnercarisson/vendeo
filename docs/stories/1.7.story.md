## User Story

**Como** equipe de engenharia do Vendeo,
**Quero** compatibilizar a leitura e a escrita de assets aprovados com a nova tabela canonica,
**Para** usar `storage_path` como verdade persistida sem quebrar campanhas ja aprovadas nem a UI atual.

---

## Objective

Entregar o ajuste minimo para que o fluxo atual de aprovacao e leitura passe a suportar `campaign_approved_assets`. Esta story deve isolar somente a compatibilidade de asset aprovado, sem misturar weekly plan, Brand Profile ou cleanup do legado.

---

## Scope IN

- Implementar resolvedor de asset aprovado que prefira `campaign_approved_assets.storage_path`.
- Usar `public_url_legacy` ou ponteiros legados apenas como fallback transitorio.
- Ajustar a escrita minima de aprovacao para suportar o registro canonico em `campaign_approved_assets`.
- Preservar os ponteiros operacionais ainda necessarios para a UI atual enquanto houver dependencia deles.

## Scope OUT

- Nao alterar renderer.
- Nao migrar UI para um fluxo novo.
- Nao mudar politica de storage do bucket atual alem do estritamente necessario para compatibilidade.
- Nao executar cleanup definitivo de `campaigns.image_url`.

---

## Acceptance Criteria

```gherkin
GIVEN um asset aprovado com registro em `campaign_approved_assets`
WHEN a aplicacao resolver o asset para exibicao ou acesso
THEN `storage_path` e usado como fonte de verdade persistida
AND a URL legada so e usada quando nao houver path canonico disponivel
```

```gherkin
GIVEN o fluxo atual de aprovacao de campanha
WHEN a compatibilidade desta story estiver ativa
THEN o registro canonico de aprovacao passa a ser suportado
AND a UI atual continua funcional para campanhas existentes e novas
```

### Implementation Notes

- A preferencia canonica de leitura foi centralizada em `lib/supabase/storage-server.ts`: quando existe registro ativo em `campaign_approved_assets`, a exibicao passa a resolver primeiro `storage_path` + `storage_bucket`; `public_url_legacy` e o ponteiro legado da campanha ficam como fallback transitorio.
- A escrita minima da aprovacao foi encapsulada em server actions autenticadas em `lib/supabase/storage-actions.ts`, que buscam a campanha atual, marcam assets aprovados anteriores do mesmo `asset_kind` como `superseded` e inserem o novo registro canonico com `campaign_snapshot`, `brand_profile_version` e `brand_profile_snapshot`.
- Os fluxos atuais de aprovacao de arte em `NewCampaignShell` e `CampaignPreviewClient` continuam atualizando `campaigns` para manter a UI operacional, mas novas aprovacoes agora persistem `campaigns.image_url` como `storage_path` e tambem registram `campaign_approved_assets`.
- A UI atual foi mantida: lista de campanhas, modal de arte e pagina de detalhe continuam consumindo os mesmos componentes, mas passam a usar o resolvedor canonico antes de cair no legado.

### Issues Encountered

- Nao ha transacao unica envolvendo `campaigns` e `campaign_approved_assets` no fluxo atual baseado em client components + upload em storage; a mitigacao aplicada foi manter `campaigns.image_url` como ponteiro operacional e falhar explicitamente se o registro canonico nao puder ser gravado.

### Observations

- Aderencia estrita ao escopo: a story ficou concentrada em compatibilidade de approved assets, sem introduzir renderer novo, UI nova ou cleanup estrutural do legado.
- `storage_path` esta corretamente tratado como fonte de verdade persistida: os dois fluxos reais de aprovacao atualizam `campaigns.image_url` com o path no bucket e tambem gravam o registro canonico em `campaign_approved_assets`.
- O fallback legado ficou restrito ao transitório: o resolvedor central sempre tenta `storage_path`, depois `public_url_legacy` e, apenas se nada existir no registro canonico, usa o ponteiro legado da campanha.
- A UI atual permanece compatibilizada por reuse dos mesmos componentes: lista, modal e pagina de detalhe agora consultam o resolvedor canonico antes de cair no legado, o que cobre campanhas novas e antigas sem trocar o fluxo visual.
- A validacao manual minima foi concluida: approve novo funcionando, campanha antiga sem registro canonico abrindo corretamente via fallback, e reaprovação mantendo somente o registro mais recente como `approved` e o anterior como `superseded`.
