## User Story

**Como** equipe de engenharia do Vendeo,
**Quero** consolidar todas as funções puras de campanha em `selectors.ts`, eliminando duplicatas e divergências entre `logic.ts` e `selectors.ts`,
**Para** ter uma única fonte de verdade para seletores, eliminar a confusão sobre qual arquivo usar e melhorar testabilidade.

---

## Objective

Migrar as 7 funções únicas de `logic.ts` para `selectors.ts`, resolver 5 conflitos/divergências com decisões documentadas, deprecar `logic.ts` como arquivo wrapper com re-exports, e criar `selectors.test.ts` com CAMPAIGN_FIXTURE completa cobrindo todos os selectors.

### Notas de equivalência (calculateGlobalStatus vs getGlobalStatus)

`calculateGlobalStatus` usa `Math.min` de scores: `none=3, draft=0, ready=1, approved=2`.  
`getGlobalStatus` usa `statuses.includes("draft")` → `"ready"` → `"approved"`.

**Potencial divergência:** Para `post_status="none"` + `reels_status="draft"` + `campaign_type="both"`:
- `calculateGlobalStatus`: `min(3, 0) = 0` → `"draft"` ✓
- `getGlobalStatus`: só inclui statuses para os tipos ativos (`"both"` inclui ambos) → `["none", "draft"]` → `"draft"` ✓

Para `post_status="ready"` + `reels_status="approved"` + `campaign_type="both"`:
- `calculateGlobalStatus`: `min(1, 2) = 1` → `"ready"` ✓
- `getGlobalStatus`: `["ready", "approved"]` → `"ready"` ✓ (includes "ready" antes de approved)

Algoritmos parecem equivalentes para inputs válidos. Confirmar com testes antes de deprecar.
