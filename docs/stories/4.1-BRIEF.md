## 🎯 Objetivo

Criar o **Motor 1 (Visual Reader v2)** da pipeline de composição visual da Epic 4.

O Visual Reader recebe uma imagem de produto e retorna um perfil visual estruturado (`ImageProfile`) que será consumido pelo Motor 2 (Intent Resolver, Story 4.2) para tomar decisões de composição.

---

## 📦 Contexto do Produto

O Vendeo gera artes para campanhas de lojas físicas. O lojista:
1. Escolhe uma aba: **Produtos** | **Serviços** | **Informativos**
2. Digita o nome do produto (ex: "Coca Cola 600ml")
3. Faz upload de uma imagem

O Visual Reader analisa essa imagem e responde: "o que tem aqui, onde está e como está?"

Essa resposta guia todos os motores seguintes da pipeline.

---

## 🏗️ Decisão de Arquitetura

### Implementação nova vs existente

O projeto já tem um Visual Reader em sandbox (`lib/visual-reader/`). Ele funciona bem como referência, mas o contrato foi redesenhado.

**Decisão:**
- Criar implementação nova em `lib/ai/visual-reader/` com o contrato definido neste documento
- Manter `lib/visual-reader/` intocado como referência durante desenvolvimento
- Sandbox (`app/api/sandbox/visual-reader/crop/`) continua apontando para o código antigo durante o desenvolvimento
- Após aprovação do novo: remover `lib/visual-reader/` e a rota sandbox

### Integração com pipeline de produção
- Criar rota de produção (não sandbox) para o novo motor
- O novo motor será chamado pela pipeline de geração de campanhas

### Modelo de IA
- **GPT-5.4 mini** — modelo em uso no sandbox atual, validado como funcional
- Multimodal (vision) com forte seguimento de instruções
- Temperature: 0.2
- Timeout: 25s
- Avaliação de upgrade para GPT-5.4 completo postergada para Story 4.1.1 (só se houver falhas de precisão no `targetBox`)

### Cache
- Supabase (baseline) — mesma imagem não deve ser analisada duas vezes
- Meta de latência: < 2s sem cache, < 500ms com cache

## 📥 CONTRATO DE ENTRADA (3 campos)

| Campo | Tipo | Obrigatório | Origem |
|-------|------|-------------|--------|
| `imageUrl` | string (URL) | ✅ | Upload do usuário → Supabase Storage |
| `productName` | string | ✅ | Digitado pelo usuário no formulário |
| `content_type` | "product" \| "service" \| "message" | ✅ | Derivado da aba ativa no formulário |

### Regra crítica de entrada
Se `content_type === "message"` → **não chamar a API de IA**. Retornar output vazio imediatamente.  
Motivo: campanhas informativas não têm produto para localizar — processar seria desperdício e geraria output sem sentido.

### Sobre `productName`
- O motor deve interpretar o nome diretamente, sem derivações externas
- Nome pode ser simples: "Coca Cola 600ml"
- Nome pode ser composto: "Coca Cola 600ml + Batatas Fritas"
- Nome pode indicar kit: "Kit c/6 Coca Cola 600ml" ou "5 pares de meias sociais"
- O motor infere o tipo de cena a partir do nome + imagem (não recebe essa informação como entrada)

## 📤 CONTRATO DE SAÍDA (21 campos)

### Grupo 1 — Match Detection (7 campos)

| Campo | Tipo | Para que serve |
|-------|------|----------------|
| `detected` | boolean | Produto exato encontrado? |
| `matchType` | "exact" \| "category_only" \| "none" | Nível de correspondência com o productName |
| `matchedTarget` | string / null | Descrição do que foi encontrado na imagem |
| `confidence` | "low" \| "medium" \| "high" | Confiança geral da análise |
| `sceneType` | "single_product" \| "multiple_products" \| "lifestyle_scene" \| "full_scene" \| "unclear" | Tipo de cena |
| `relevantCount` | number | Quantos itens relevantes foram encontrados |
| `ignoredElements` | string[] | Itens presentes na imagem que não são o alvo |

**Comportamento do `matchType`:**
- `exact` → produto encontrado com marca, formato e volume correspondentes
- `category_only` → encontrou produto similar (mesma categoria, marca ou formato diferente)
- `none` → produto não encontrado na imagem

**Sobre `ignoredElements`:**  
Itens ignorados têm valor futuro para cross-sell — o Motor 2 poderá usá-los se esses itens fizerem parte do mix da loja. Manter sempre que detectados.

**Comportamento do `confidence`:**  
Nunca bloqueia a geração. O sistema alerta o usuário sobre baixa confiança, mas a decisão de publicar é sempre do lojista.

### Grupo 2 — Spatial (4 campos)

| Campo | Tipo | Para que serve |
|-------|------|----------------|
| `targetBox` | {x, y, width, height} / null | Coordenadas normalizadas (0-1) do produto na imagem |
| `targetOrientation` | "vertical" \| "horizontal" \| "square" \| "mixed" \| "unknown" | Orientação do produto |
| `targetPosition` | "left" \| "center" \| "right" \| "top" \| "bottom" \| "mixed" \| "unknown" | Posição do produto na imagem |
| `targetOccupancy` | "low" \| "medium" \| "high" \| "full" | Quanto da imagem o produto ocupa |

**Sobre `targetBox`:**
- Coordenadas normalizadas: x e y entre 0 e 1 (proporção da imagem)
- Quando `productName` é composto ("Coca + Batatas"): retornar **uma box única** que engloba todos os itens
- `null` apenas quando `matchType === "none"`

**Sobre `targetPosition` e `targetOccupancy`:**
- Servem de fallback quando `targetBox` é null
- `targetOccupancy` indica margem de manobra para o Motor 2: `low` = muito espaço disponível, `full` = sem espaço

### Grupo 3 — Quality (3 campos)

| Campo | Tipo | Para que serve |
|-------|------|----------------|
| `imageQuality` | "good" \| "acceptable" \| "poor" \| "unknown" | Qualidade técnica geral da foto |
| `visibility` | "clear" \| "partial" \| "obstructed" \| "unknown" | Produto visível? Obstruído? Cortado? |
| `framing` | "good" \| "tight" \| "distant" \| "unknown" | Enquadramento: produto muito próximo, distante ou adequado? |

**Diferença entre `visibility` e `framing`:**
- `visibility`: a Coca está coberta por outra garrafa na frente? → `partial`
- `framing`: a Coca está muito pequena no canto da foto? → `distant`

Ambos informam o usuário sobre a adequação da foto e guiam o Motor 2 na composição.

**Alertas ao usuário** (não bloqueiam, apenas informam):
- `imageQuality === "poor"` → foto de baixa qualidade
- `visibility === "obstructed"` → produto encoberto
- `framing === "distant"` → produto muito pequeno
- `matchType === "none"` → produto não encontrado
- `matchType === "category_only"` → produto diferente do declarado

### Grupo 4 — Background (1 campo)

| Campo | Tipo | Para que serve |
|-------|------|----------------|
| `backgroundType` | "transparent" \| "solid" \| "complex" \| "unknown" | Tipo de fundo — decisão principal do Motor 2 |

**O que cada valor significa para a composição:**
- `transparent` → imagem sem fundo (PNG com alpha). Pode ser usada livremente em qualquer composição
- `solid` → fundo de cor sólida. Fácil de manipular — Motor 2 pode substituir ou manter
- `complex` → ambiente, loja, cena de uso. Usar imagem inteira — não tentar recortar o produto

### Grupo 5 — Crop Potential (4 campos — temporários)

> Estes campos são mantidos para coleta de dados e testes. Serão reavaliados na Story 4.1.1 após volume real de imagens.

| Campo | Tipo | Para que serve |
|-------|------|----------------|
| `subjectCutoff` | "none" \| "light" \| "moderate" \| "severe" \| "unknown" | Produto cortado nas bordas da foto? |
| `safeExpansionPotential` | "low" \| "medium" \| "high" \| "unknown" | Espaço ao redor do produto para expandir crop |
| `focusClarity` | "low" \| "medium" \| "high" \| "unknown" | Foco específico no produto (diferente de imageQuality que avalia a foto toda) |
| `visualIsolation` | "low" \| "medium" \| "high" \| "unknown" | Produto destacado do fundo? (produto pode estar em fundo sólido mas com cor similar) |

### Grupo 6 — Metadata (1 campo — temporário)

| Campo | Tipo | Para que serve |
|-------|------|----------------|
| `reasoningSummary` | string | Explicação em texto do que o modelo encontrou. Útil para debug durante testes. Sem uso nos motores ou no usuário. |

> Candidato a remoção na Story 4.1.1.

## 🔔 Alertas ao Usuário

O sistema **nunca bloqueia** a geração. Apenas informa. O lojista decide.

| Condição | Mensagem sugerida |
|----------|------------------|
| `matchType === "none"` | "Produto não encontrado na imagem. Verifique se a imagem corresponde ao produto." |
| `matchType === "category_only"` | "A imagem parece ter um produto diferente do declarado (`matchedTarget`). Deseja continuar?" |
| `imageQuality === "poor"` | "A qualidade da imagem está baixa. O resultado pode não ser o ideal." |
| `visibility === "obstructed"` | "O produto parece estar parcialmente encoberto na imagem." |
| `framing === "distant"` | "O produto está muito pequeno na imagem. Tente uma foto mais próxima." |
| `confidence === "low"` | "A análise da imagem teve baixa confiança. Revise o resultado." |

## 🤝 Dependência de Agente — @prompt-eng

**O `@dev` deve chamar o `@prompt-eng` como primeira tarefa da story**, antes de escrever qualquer código de implementação.

O `@prompt-eng` é responsável por compor o system prompt do `gpt-5.4-mini` (vision) que será salvo em `lib/ai/visual-reader/prompts.ts`.

### O que entregar ao @prompt-eng

Passar este documento como contexto completo. Os pontos críticos que o prompt deve cobrir:

**Objetivo do motor:**
- Analisar uma imagem e localizar o produto descrito em `productName`
- Retornar um JSON estruturado com os 21 campos do contrato de saída
- Nunca inventar — basear-se apenas no que é visível na imagem

**Regras de matching:**
- `exact` → marca + formato + volume correspondem ao `productName`
- `category_only` → produto similar encontrado, mas não é o declarado — retornar `targetBox` e descrever o que foi encontrado em `matchedTarget`
- `none` → produto não encontrado — `targetBox` deve ser null

**Regras de `productName` composto:**
- Nome com "+" ou "kit" ou "pack" ou "c/N" indica múltiplos itens
- Localizar todos os itens do nome na imagem
- Retornar uma `targetBox` única que englobe todos eles
- Se itens estiverem em extremos opostos da imagem, a box será grande — isso é correto

**Regras de `targetBox`:**
- Coordenadas normalizadas entre 0 e 1 (proporção da imagem)
- Deve ser justa — envolver apenas o produto, não a imagem inteira
- Obrigatória para `matchType=exact` e `matchType=category_only`
- `null` apenas para `matchType=none`

**Regras de `backgroundType`:**
- `transparent` → PNG sem fundo / alpha channel visível
- `solid` → fundo de cor única e uniforme
- `complex` → ambiente, cena, múltiplos objetos ao fundo

**Formato de resposta:**
- JSON puro e válido
- Sem markdown, sem texto antes ou depois
- Sem campos extras além dos 21 definidos

### O que o @prompt-eng deve entregar

- System prompt final para `lib/ai/visual-reader/prompts.ts`
- Justificativa das escolhas de estruturação do prompt
- Exemplos de few-shot (se recomendado)
