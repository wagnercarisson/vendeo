## 🎯 Problem

### Current Behavior
Motor 3 (Visual Composer) generates **100+ Zod validation errors** per request:

```
[MOTOR-3][FALLBACK] Zod validation failed for output 1, using fallback structure
[MOTOR-3][ERROR] Zod validation error for output 1:
```

## 🔍 Root Cause Hypotheses

### H1: Schema Drift (Most Likely — 🔴 HIGH)
GPT-5.4 evolved and now returns fields not anticipated in original schema:
- `productName` in `compositions[].productName`
- `priceBadge` in `compositions[].elements[]`
- Numeric types where schema expects strings (`fontWeight: 700` vs `'700'`)
- Additional font properties (`fontFamily`, `lineHeight`)

**Likelihood:** 🔴 HIGH — Evidence shows consistent field mismatches  
**Validation:** Analyze 10+ production outputs, identify consistent fields

### H2: Prompt Instruction Mismatch (🟡 MEDIUM)
Motor 3 prompt doesn't explicitly align with Zod schema (no JSON schema in prompt).

**Likelihood:** 🟡 MEDIUM — Prompt may allow creative freedom  
**Validation:** Review prompt, check if schema constraints mentioned

### H3: Schema Overly Strict (🟡 MEDIUM)
Schema designed conservatively, rejecting useful fields GPT wants to provide.

**Likelihood:** 🟡 MEDIUM — Schema may need loosening  
**Validation:** Assess if rejected fields improve composition quality

## 🏗️ Solution Approach — **OPTION C: HYBRID** (Chosen)

**Based on investigation findings, Option C provides optimal balance:**

### Implementation Strategy

**1. Accept Useful Fields in Schema**
```typescript
// lib/ai/visual-composer/contracts.ts
export const TypographySpecSchema = z.object({
  fontSize: z.number().int().positive(),
  fontWeight: z.coerce.string().pipe(z.enum(["400", "600", "700", "900"])), // Coerce number→string
  // NEW FIELDS (optional):
  fontFamily: z.string().optional(),
  color: z.string().regex(/^#[0-9A-Fa-f]{6}$/).optional(),
  lineHeight: z.number().positive().optional(),
});
```

**2. Fix Type Issues**
- `fontWeight`: Add `.coerce` to convert number (700) → string ("700")
- Remove `.strict()` mode (use default: validates defined fields, ignores extras)

**3. Constrain Prompt to Exclude Noise**
```typescript
// lib/ai/visual-composer/service.ts (prompt update)
"Do NOT include these fields:
- promotion object
- renderingHints object  
- debugMetadata object
- targetConstraints object

ALWAYS include these fields:
- fontFamily in all typography objects
- color in all typography objects
- lineHeight for professional spacing"
```

### Expected Outcomes

| Metric | Before | After | Delta |
|--------|--------|-------|-------|
| Validation Errors | 20-32 | **0** | ✅ -100% |
| Fallback Rate | 100% | **0%** | ✅ Eliminated |
| Performance | 23s | **20s** | ✅ -13% |
| Log Noise | 400+ lines | **<50 lines** | ✅ -87% |
| Output Quality | Generic | **Professional** | ✅ Better |

**Effort:** 4-5 hours (schema update + prompt refinement + tests)

**Risk:** LOW — All fields optional, backward compatible, fallback preserved

### Why NOT Option A or B Alone

**Option A alone:** Accepts all GPT fields without constraint → noise fields persist (20-40% frequency)  
**Option B alone:** Constrains GPT but doesn't fix type issues (fontWeight) or schema strictness

**Option C combines:** Accept useful fields + fix types + constrain noise = best outcome
