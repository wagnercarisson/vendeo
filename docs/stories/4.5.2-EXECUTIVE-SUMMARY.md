## Problem (Current State)

Motor 3 (Visual Composer) generates valid composition layouts but **fails validation 100% of the time**:

```
✅ GPT generates rich, useful output
❌ Zod schema rejects extra fields → Fallback generation
❌ Motor 4 receives generic output (no fonts, colors, styling)
⏱️ Performance: 23 seconds (15% overhead from validation fail + fallback)
```

**Root Cause:** Schema is too strict (`.strict()` mode). GPT evolved, schema didn't.

---

## Analysis Results

### Fields Identified (10 total)

| Category | Fields | Impact | Frequency |
|----------|--------|--------|-----------|
| **ÚTIL** (Keep) | fontFamily, color, lineHeight, productName | HIGH | 80-100% |
| **CONFLITO** (Fix) | fontWeight (type: number vs string) | MEDIUM | 100% |
| **RUÍDO** (Remove) | promotion, renderingHints, debugMetadata | LOW | 20-60% |

### Key Finding

**GPT is NOT hallucinating.** Fields are legitimate, useful enhancements:
- `fontFamily`: Motor 4 needs this for typography rendering
- `color`: Without it, text renders black (generic)
- `lineHeight`: Professional spacing (currently ignored)

---

## Recommendation: OPTION C (Hybrid)

### What to Do

1. **Update Schema** (contracts.ts)
   - Add fontFamily, color, lineHeight as optional fields
   - Fix fontWeight type: coerce number → string

2. **Constrain Prompt** (prompts.ts)
   - Instruct GPT to NOT return promotion, renderingHints, etc.
   - Keep instructions for useful fields

3. **Zero Impact to Motor 4**
   - Already receives Motor 3 output
   - New fields automatically available for rendering
   - No breaking changes

### Expected Outcomes

| Metric | Before | After | Delta |
|--------|--------|-------|-------|
| Validation Errors | 20-32 | 0 | ✅ -100% |
| Fallback Rate | 100% | 0% | ✅ Eliminated |
| Performance | 23s | 20s | ✅ -13% |
| Output Quality | Generic | Professional | ✅ Better |

## Decision Point

### Option A (Not Recommended)
Update schema to accept ALL fields from GPT (including noise)
- ❌ Adds maintenance burden
- ✅ Simple one-off fix
- ❌ Doesn't solve problem of extra fields

### Option B (Rejected)
Constrain prompt only, reject schema changes
- ✅ Minimal code change
- ❌ Doesn't leverage useful fields
- ❌ Leaves Motor 4 with degraded output

### Option C (RECOMMENDED) ✅
Accept useful fields + constrain prompt + fix types
- ✅ Clean separation: fields (accept) vs noise (reject)
- ✅ Maximizes Motor 4 output quality
- ✅ Balanced approach, lowest risk
- ✅ Future-proof as GPT evolves further
