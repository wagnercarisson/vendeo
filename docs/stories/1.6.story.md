## User Story

**Como** equipe de engenharia do Vendeo,
**Quero** adaptar o fluxo de weekly plan para carregar e promover a intencao de dominio planejada,
**Para** separar corretamente planejamento e execucao sem quebrar o weekly plan atual.

---

## Objective

Entregar a menor compatibilidade possivel para weekly plan apos a existencia do schema novo e da normalizacao de dominio de campanha. Esta story deve apenas conectar os campos de planejamento ao fluxo atual sem misturar assets, marca ou cleanup.

---

## Scope IN

- Ajustar tipos, schemas, mappers e services de weekly plan para carregar `target_content_type` e `target_domain_input`.
- Preservar `weekly_plan_items.content_type` como formato operacional legado.
- Ajustar a promocao minima para campanha para copiar `target_content_type` e `target_domain_input` quando aplicavel.

## Scope OUT

- Nao fazer backfill de planejamento antigo.
- Nao alterar approved assets.
- Nao alterar fallback de Brand Profile.
- Nao renomear `weekly_plan_items.content_type`.
- Nao criar nova UI de planejamento.

---

## Acceptance Criteria

```gherkin
GIVEN um item de weekly plan com `target_content_type` e `target_domain_input`
WHEN esse item for carregado pela camada de dominio
THEN a intencao de dominio planejada fica disponivel sem sobrescrever o formato legado
```

```gherkin
GIVEN a promocao oficial de um item de plano para campanha
WHEN o fluxo compatibilizado executar
THEN `campaigns.content_type` recebe `target_content_type` quando presente
AND `campaigns.domain_input` recebe `target_domain_input` com as normalizacoes server-side minimas
```

### Implementation Notes

- `WeeklyPlanItem` passou a expor explicitamente dois eixos: `content_type` legado de formato operacional (`post | reels`) e a intencao planejada via `target_content_type` + `target_domain_input`.
- O dominio de `weekly-plans` agora faz leitura tolerante e normalizacao minima de `target_content_type`/`target_domain_input`, sem reinterpretar `weekly_plan_items.content_type`.
- A geracao de novos itens do plano preserva `content_type` legado e carrega `target_*` quando esses campos existirem na estrategia aprovada, sem exigir backfill nem UI nova.
- A promocao para campanha via `plan_item_id` passou a buscar o item do plano, usar `target_content_type` como fonte preferencial de dominio e preencher `campaigns.domain_input` com normalizacao minima a partir de `target_domain_input` e dos dados editados no fluxo de campanha.
- Os atalhos de orquestracao do weekly plan passaram a enviar `type` separado do `content_type` legado, evitando inferencia de dominio a partir de `post`/`reels`.

### Observations

- Aderência estrita ao escopo: a story ficou limitada ao domínio de `weekly-plans`, aos atalhos de orquestração e à promoção mínima para campanha, sem puxar approved assets, Brand Profile, rename de coluna ou UI nova.
- Separação semântica correta: `weekly_plan_items.content_type` segue como formato operacional legado, enquanto `target_content_type` e `target_domain_input` carregam a intenção planejada sem colisão de significado.
- Promoção mínima correta: a campanha derivada de plano não infere mais domínio a partir de `post` ou `reels`; ela usa `target_content_type` quando disponível e monta `domain_input` com preenchimento mínimo a partir do target e dos dados editados.
- Compatibilidade do fluxo atual preservada: os componentes de plano continuam funcionando com os atalhos existentes, apenas adicionando `type` quando há domínio planejado explícito, o que reduz risco de regressão na orquestração atual.
- Risco residual não bloqueante: a validação automatizada permaneceu apoiada em `npx tsc --noEmit` e inspeção dos fluxos, porque o repositório ainda não tem suite dedicada para promoção `weekly_plan_item -> campaign`.
