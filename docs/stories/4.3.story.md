## 📋 Context

O Vendeo gera artes para campanhas de lojas físicas. Após análise da imagem (Motor 1 - Visual Reader) e definição da direção criativa (Motor 2 - Intent Resolver), o sistema precisa **compor variações de layout** para o usuário escolher.

O **Visual Composer v2** é o **Motor 3** da pipeline de composição visual. Ele recebe todos os contextos dos motores anteriores e responde: "onde cada elemento vai ficar na arte?" — retornando 4 variações de layout estruturadas (`CompositionSpec`) com posições absolutas em pixels.

**Source of Truth:** [4.3-BRIEF.md](./4.3-BRIEF.md)

**Critical Dependencies:**
- Story 4.1 (Visual Reader) MUST be complete
- Story 4.2 (Intent Resolver) MUST be complete

## 🎯 Goal

Implementar o **Motor 3 (Visual Composer)** com:
- Contrato de entrada: 27 campos agregados (ImageProfile + CreativeDirection + VisualSignature + CampaignData)
- Contrato de saída: 4 variações de layout (`CompositionSpec`) com ~40 valores cada (~160 valores totais)
- Canvas fixo: 1080×1350 pixels (MVP beta)
- Validação: bounds, overlap, 30% distinctness entre variações
- System prompt otimizado via @prompt-eng
- Fallback determinístico se LLM falhar
- 12 fixtures de teste cobrindo cenários críticos

## 🏗️ Technical Approach

### Architecture Decision
- **New implementation:** `lib/ai/visual-composer/` com novo contrato
- **No legacy code:** Esta é a primeira implementação (não há versão anterior)
- **Production ready:** Deve ser usado diretamente em produção após validação

### AI Model
- **gpt-5.4** (full, NÃO mini)
- Temperature: 0.4 (mais criativo que Motor 2 para gerar variações distintas)
- Timeout: 25s (mais tempo que Motor 2 — output maior)
- Justificativa: Raciocínio espacial complexo + gerar 4 variações distintas em 1 chamada

### Layout Strategy
- Coordenadas em **pixels absolutos** (0-1080 × 0-1350)
- Baseado em `directionType` do Motor 2 (hero, frame, split-dynamic, overlay, stacked)
- 4 áreas: `productArea`, `textArea`, `priceArea`, `badgeArea` (opcional)
- Validação automática: bounds, overlap (exceto overlay), distinctness 30%

## 🔒 Validation Rules

### Rule 1: Canvas Boundaries

All elements MUST be within canvas 1080×1350:

```typescript
function validateBounds(area: {x, y, width, height}): boolean {
  return (
    area.x >= 0 &&
    area.y >= 0 &&
    area.x + area.width <= 1080 &&
    area.y + area.height <= 1350
  );
}
```

---

### Rule 2: No Overlap (elements do not overlap)

**Except when `textDistribution === "overlay"`** (intentional overlap).

```typescript
function areasOverlap(a1, a2): boolean {
  return !(
    a1.x + a1.width < a2.x ||
    a2.x + a2.width < a1.x ||
    a1.y + a1.height < a2.y ||
    a2.y + a2.height < a1.y
  );
}

// Validate that productArea, textArea, priceArea do not overlap
if (direction.textDistribution !== "overlay") {
  if (areasOverlap(productArea, textArea)) {
    // INVALID
  }
}
```

---

### Rule 3: 30% Distinctness Between Variations

**Ensure that the 4 variations are visually distinct.**

```typescript
function validateDistinctness(variations: CompositionSpec[]): boolean {
  const threshold = 0.3; // 30% minimum difference
  
  for (let i = 0; i < variations.length; i++) {
    for (let j = i + 1; j < variations.length; j++) {
      const v1 = variations[i];
      const v2 = variations[j];
      
      // Difference in product position (horizontal)
      const xDiff = Math.abs(v1.layout.productArea.x - v2.layout.productArea.x);
      const xDiffPercent = xDiff / 1080;
      
      // Difference in product position (vertical)
      const yDiff = Math.abs(v1.layout.productArea.y - v2.layout.productArea.y);
      const yDiffPercent = yDiff / 1350;
      
      // Difference in size
      const widthDiff = Math.abs(v1.layout.productArea.width - v2.layout.productArea.width);
      const widthDiffPercent = widthDiff / 1080;
      
      const heightDiff = Math.abs(v1.layout.productArea.height - v2.layout.productArea.height);
      const heightDiffPercent = heightDiff / 1350;
      
      // At least ONE metric must have >30% difference
      const maxDiff = Math.max(xDiffPercent, yDiffPercent, widthDiffPercent, heightDiffPercent);
      
      if (maxDiff < threshold) {
        return false; // Variations too similar
      }
    }
  }
  
  return true;
}
```

**If validation fails:** Regenerate variations (maximum 2 attempts) or use deterministic fallback.

### Execution flow

```typescript
// lib/ai/visual-composer/service.ts
export async function composeVariations(
  imageProfile: ImageProfile,
  direction: CreativeDirection,
  signature: VisualSignature,
  campaign: CampaignData
): Promise<CompositionVariants> {
  
  // 1. Gate: informative campaigns do not process
  if (campaign.content_type === "message") {
    return emptyCompositionVariants();
  }

  // 2. Prepare context for LLM
  const context = {
    image: imageProfile,
    direction,
    signature,
    campaign,
    canvas: { width: 1080, height: 1350 },
  };

  // 3. Call LLM (generates 4 variations in 1 call)
  const response = await callAI(
    [{ role: "system", content: VISUAL_COMPOSER_SYSTEM_PROMPT }],
    { role: "user", content: JSON.stringify(context) }
  );

  // 4. Parse and validation
  const parsed = parseJsonFirstObject(response);
  const validated = CompositionVariantsSchema.safeParse(parsed);

  if (!validated.success) {
    // Fallback: generate deterministic variations
    return generateFallbackVariations(context);
  }

  const variations = validated.data.variations;

  // 5. Validate distinctness
  if (!validateDistinctness(variations)) {
    // Retry once or use fallback
    return generateFallbackVariations(context);
  }

  // 6. Validate bounds and overlap
  for (const v of variations) {
    if (!validateBounds(v.layout.productArea)) {
      throw new Error("INVALID_LAYOUT_BOUNDS");
    }
    // ... validate other areas
  }

  return {
    direction,
    variations,
    canvas: { width: 1080, height: 1350 },
    generated_at: new Date().toISOString(),
  };
}
```
